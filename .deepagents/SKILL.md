# 长文调研报告写作 Skill（blog-post）

这个 skill 提供一套稳定的长文写作流程，适合把“调研结果”整理成一份结构清晰、内容扎实的报告文档。

## 什么时候用

当用户让你做以下事情时用这个 skill：

- 写调研报告 / 研究报告
- 写长文总结、教程、科普文章
- 写结构化的深度文章（需要目录、分段、论证）

## 先调研（强制）

在写正式报告之前，必须先把资料调研做完：

1. 用 `task` 工具派发调研子任务（`subagent_type: "researcher"`）
2. 在任务描述里明确：调研主题 + 需要输出哪些要点 + 让子任务把结论写成文档（用 `write_file` 工具保存）
3. 子任务完成后，再读子任务产出的文档内容，最后再写主报告

示例（仅示意流程）：

```
task(
  subagent_type="researcher",
  description="调研【主题】的现状、关键结论、数据/案例与可引用的来源。把调研结果用 write_file 保存成文档。"
)
```

## 输出产物（强制）

写作类任务必须把最终内容写成文档，而不是只在聊天框里输出：

- 最终报告：必须调用 `write_file` 保存
- 报告内容建议包含：标题、摘要、目录/结构、正文、结论与建议

注意：
- `write_file` 的 `file_path` 只是标识用，不要在回复里重复文件名/路径（按系统规则执行）

## 报告结构（建议模板）

每份调研报告建议按下面结构组织：

### 1. 摘要（开头）

- 2-5 句话说清楚：结论是什么、为什么重要、给谁看

### 2. 背景与问题定义

- 解释这个话题为什么值得调研
- 明确调研范围与边界（包含什么/不包含什么）

### 3. 核心发现（3-6 个要点）

- 每个要点一个小节（H2/H3）
- 每个要点都要：观点 + 证据/案例 + 对业务的影响

### 4. 方案/建议

- 给出可落地的建议（按优先级/阶段拆分）
- 如果有多种路线，列出对比（成本、风险、收益、实施周期）

### 5. 风险与限制

- 明确哪些信息不确定、哪些前提可能变化
- 给出后续验证方式

### 6. 结论与下一步

- 3 条以内总结
- 明确下一步动作

## 质量自检清单（写完必查）

- [ ] 是否先做了调研，再写报告
- [ ] 是否把最终报告用 `write_file` 保存了
- [ ] 结构是否清晰（标题层级、段落、要点）
- [ ] 关键结论是否有依据/案例支撑
- [ ] 是否给出可执行的建议与下一步
